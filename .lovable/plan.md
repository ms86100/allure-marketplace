

# Fix Plan: Delivery OTP Not Visible to Buyer + Profile Navigation Issue

## Issue 1: Buyer Cannot See Delivery OTP

### Root Cause — Database Trigger Race Condition
The delivery OTP is generated by a trigger (`trg_generate_delivery_code`) on the `orders` table that fires when order status changes to `picked_up`. However, for seller-delivery orders, the delivery assignment itself is ALSO created by another trigger (`trg_create_seller_delivery_assignment`) on the same `orders` UPDATE event.

Both are `AFTER UPDATE` triggers. In PostgreSQL, AFTER triggers on the same row see the same snapshot. The assignment INSERT from one trigger is **not visible** to the UPDATE in the other trigger. Result: `delivery_code` stays `NULL`.

**Evidence from live data:**
- 3 out of 4 recent delivery assignments have `delivery_code: NULL`
- The UI code at line 318 of `OrderDetailPage.tsx` correctly renders the OTP when `buyerOtp` is set — the problem is purely that `delivery_code` is never populated in the database

### Fix
Move delivery code generation INTO the `trg_create_seller_delivery_assignment` function itself, so the code is set at INSERT time — no second trigger needed. Also add a safety net: a separate trigger on `delivery_assignments` INSERT that generates the code if missing.

**Database migration:**
1. Modify `generate_delivery_code()` to be a trigger on `delivery_assignments` (INSERT) instead of `orders` (UPDATE)
2. Inside `trg_create_seller_delivery_assignment`, generate the 4-digit code inline during the INSERT
3. Keep the `delivery_assignments` INSERT trigger as a safety net for platform-assigned deliveries
4. Backfill any existing NULL codes on active assignments

### Additionally: Show OTP Earlier and More Prominently
The current UI only shows the OTP inside the live tracking section (`isInTransit && buyerOtp`). This means the buyer can't see it if the tracking section isn't rendered. Add a persistent OTP display card that appears regardless of tracking state, as long as the order has a delivery code and isn't in a terminal status.

**Files changed:**
- Database migration (new) — fix trigger race + backfill
- `src/pages/OrderDetailPage.tsx` — add persistent OTP display outside the tracking section

## Issue 2: Profile Page Redirects to Order Summary

### Root Cause
The `useBuyerOrderAlerts` hook (line 75) fires a toast with a "View" action button on every order status change. During active delivery, multiple status updates occur rapidly (`picked_up` → `on_the_way` → `at_gate`). Each toast pops up with a clickable "View" button.

On mobile, the toast overlay sits on top of the bottom navigation area. When the buyer taps "Profile" in the bottom nav, they're actually tapping the toast's "View" action button, which executes `window.location.hash = '#/orders/${orderId}'` — redirecting them to Order Summary.

### Fix
1. Reposition toasts so they don't overlap the bottom navigation bar (add bottom padding/offset)
2. Make delivery status update toasts non-interactive during transit — remove the "View" action button for statuses that occur during active delivery (the buyer is already tracking the order)
3. Auto-dismiss transit toasts faster (3s instead of 6s) to reduce interference

**Files changed:**
- `src/hooks/useBuyerOrderAlerts.ts` — suppress "View" action for transit statuses, reduce duration
- Toast configuration — ensure toasts render above the bottom nav safe area
